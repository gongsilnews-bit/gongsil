# 💡 아이디어 10: AI 에이전트(페이퍼클립) 도입을 통한 공실뉴스 완전 자동화 기획안

## 1. 개요 및 목표
- **목표**: 오픈소스 AI 에이전트 오케스트레이션 플랫폼인 **페이퍼클립(paperclipai/paperclip)**을 도입하여 공실뉴스 플랫폼의 기사 작성, 매물 검수, 회원 관리 등을 '무인(Zero-human)'에 가깝게 자동화하는 아키텍처 구축.
- **배경**: 현재 구축 중인 튼튼한 프론트엔드/백엔드 아키텍처 위에, 대표자(사용자)가 큰 방향성만 지시하면 세부 업무를 전문화된 AI 에이전트 봇들이 스스로 나누어 실행하게 만듦.

## 2. AI 컴퍼니 조직도 및 역할 (Agent 역할 분담)
페이퍼클립을 통해 아래와 같은 가상 직원들을 생성하고 업무를 위임합니다.

1. **AI 편집장 (CEO Agent)**
   - 플랫폼의 전반적인 방향성 및 콘텐츠 스케줄을 총괄.
   - 대표의 지시(예: "이번 주는 강남구 재건축 이슈를 메인으로 해")를 하위 에이전트에게 할당하고 최종 승인(결재).
2. **AI 부동산 전문 기자 (Writer Agent)**
   - 자동 스케줄러(Heartbeat)에 따라 국토교통부, 주요 언론사 데이터를 수집.
   - 공실뉴스 톤앤매너에 맞춘 기사(텍스트+이미지)를 주기적으로 자동 작성 및 Supabase DB에 인서트.
3. **AI 가입 심사관 (Verification Agent) ⭐ 핵심**
   - **기능**: 회원 가입 시 첨부된 '중개사무소 등록증' 또는 '사업자등록증' 이미지를 Vision AI(GPT-4o Vision 등)로 판독.
   - **처리**: 추출한 텍스트(상호, 대표자명, 등록번호)가 폼에 입력된 정보와 일치할 경우 자동으로 `approved` 처리. 불일치 및 흐린 사진은 `pending` 처리 후 인간 관리자에게 검수 요청 알림.
4. **AI 프론트엔드 개발자 (Dev Agent)**
   - UI 수정 요청, 배너 교체 등을 인식하여 GitHub 브랜치에 코드를 기여(Commit/Push)하고 갱신.

## 3. 기술 아키텍처 및 연동 방안
페이퍼클립은 백엔드 서버(Node.js 등) 환경이 필요하므로 현재의 GitHub Pages(정적 호스팅)에 직접 이식할 수 없습니다. 따라서 다음과 같은 분리 연동 방안을 사용합니다.

- **서버 분리 배치**: Paperclip 서버는 Vercel, AWS, Heroku 등의 별도 클라우드 플랫폼에서 구동. (`admin.paperclip.gongsilnews.com`)
- **Supabase 연결**: 에이전트들이 직접 공실뉴스의 Supabase API 스키마에 접근하여 Read/Write 할 수 있는 보안 엑세스 권한 부여.
- **최고관리자 페이지(admin.html) 통합**: 
  - 관리자 좌측 메뉴(Sidebar)에 `🤖 AI 운영본부` 메뉴 신설.
  - 클릭 시 우측 메인 영역에 Paperclip 대시보드 화면을 `<iframe>`으로 삽입하여, 관리자가 하나의 웹페이지를 떠나지 않고 전체 AI 직원 통제.

## 4. 단계별 도입 로드맵 (Roadmap)
- **Phase 1: 수동 MVP 파운데이션 완성 (최우선 과제)**
  - 현재 진행 중인 공실뉴스의 모든 UI/UX, 백엔드 연동, 로그인, 수동 관리자 페이지를 완벽하게 정상 구동되도록 마무리.
  - (AI가 시스템 구조를 헷갈리지 않도록 견고한 뼈대 마련 목적)
- **Phase 2: 단일 AI 스크립트 작성 및 테스트**
  - 회원가입 이미지 비전(Vision) 판독 스크립트(Python 등) 단품 개발 및 테스트.
  - 뉴스 기사 생성 스크립트 구축 및 주기적 API 호출 테스트.
- **Phase 3: 페이퍼클립 투입 및 전면 위임**
  - Phase 2에서 만든 단품 코드들을 페이퍼클립 에이전트의 '스킬(Skill)'로 이식.
  - 에이전트 간 계층 구조 확립, 과도한 API 호출 및 요금 폭탄을 방지하기 위한 예산 한도(Budget Control) 설정 후 운용 개시.

## 5. 결론
공실뉴스의 수동 인프라 구축이 최종 완성된 후, 페이퍼클립을 통해 자동 운용 계층(Operation Layer)을 덮어씌움으로써, 대표자 1인이 대규모 부동산 포털을 운영할 수 있는 궁극적 자동화 플랫폼으로 진화할 수 있습니다.
